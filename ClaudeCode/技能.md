# Claude Code 技能（Skills）

## 一、使用场景

将"高频、规范化、有固定流程"的任务（SOP）进行持久化并命名以被 **用户或 Claude** 调用

## 二、组成结构

典型目录如下：

```
.claude/skills/<skill-name>/
├── SKILL.md                # 必需：核心入口，YAML frontmatter + Markdown 正文
├── references/             # 可选：上下文按需注入的参考文档
│   ├── coding-standards.md
│   └── api-conventions.md
├── scripts/                # 可选：可执行的辅助脚本
│   ├── lint.sh
│   └── notify.sh
├── templates/              # 可选：输出模本
│   ├── output-doc.md
├── examples/               # 可选：示例
│   ├── code-example.md
└── assets/                 # 可选：模板、图片等静态资源
    ├── review-template.md
    └── severity-icon.png
```

## 三、SKILL.md

### 3.1 文档结构

SKILL.md = YAML frontmatter + Markdown 正文

frontmatter 的所有字段均可选

```yaml
---
name: generate-api-doc
# 【技能名称】与目录名相同，最多 64 个字符，必须是 kebab-case 格式，最好：动词 + 名词
# 若省略，默认使用目录名，对应 /<name> 调用

description: Generate API documentation from Express, FastAPI, or Spring Boot source code. Use when user asks to "write API docs", "document endpoints", "create OpenAPI specs", or mentions "Swagger". Supports route detection, request/response schema extraction, and authentication requirement marking.
# 【技能描述】模型是否调用此技能的唯一依据
# 格式1：功能定义（做什么） + 触发时机（何时用） + 核心功能（能干啥），要求适合AI阅读
# 格式2：功能定义（做什么） + 核心功能（能干啥），要求适合AI阅读，将触发时机（何时用）放到 when_to_use 
# 若省略，默认使用正文的第一段
# 组合的 description 和 when_to_use 文本在 skill 列表中被截断为 1,536 个字符以减少上下文使用

when_to_use: 
# 【何时调用】关于 Claude 何时应该调用该 skill 的额外上下文，例如触发短语或示例
# 附加到 skill 列表中的 description，并计入 1,536 个字符的上限

argument-hint: [diff-path]
# 参数占位提示

disable-model-invocation: true
# 【禁止模型调用】置为 true 以防止 Claude 自动加载此 skill。
# 用于手动触发 /name ，防止该 skill 被预加载到上下文中
# 默认值：false

user-invocable: false
# 【用户调用】置为 false 以从 / 菜单中隐藏
# 用于不应直接调用的背景知识，但 Claude 会自动调用
# 默认值：true

allowed-tools:
  - Read
  - Grep
  - Glob
  - Write
  - Bash(python:*)
# 【工具白名单】精确控制 skill 执行时可调用的工具及权限范围
# 格式：接受空格分隔的字符串或 YAML 列表

model: haiku
# 【调用模型】skill 使用的模型
# 接受与 /model 相同的值，或 inherit 以保持活动模型

context: fork
# 【执行上下文】设为 fork 时，将在隔离的子智能体重执行
# 当不想污染主对话上下文时可设置

agent: Explore
# 【子智能体】当 context = fork 时生效
# 可选值：Explore、Plan、general-purpose 或自定义 subagent
---

[Markdown 正文]
```

### 3.2 撰写原则

#### 路由转发

将业务知识按类分文件放到 references 中，正文中按业务场景描述调用时机，有两种形式：速查表，case 列
```markdown
……
## 速查表
| 分析类型 | 何时使用 | 应用 |
| --- | --- | --- |
| 营收分析 | 收入、营收、销售额 | `references/revenue.md` |
| 成本分析 | 成本、费用、支出 | `references/costs.md` |
| 盈利分析 | 利润、毛利率、净利率 | `references/profitability.md` |
……
```

```markdown
……
## 步骤3：计算指标
公式与计算：
- 营收指标 → 参考 `references/revenue.md`
- 成本指标 → 参考 `references/costs.md`
- 盈利指标 → 参考 `references/profitability.md`
……
```

#### 契约式引用

在正文中引用文件时，要按照契约书写：触发时机 + 资源位置 + 预期产出，模板：当 …… 时，加载：……，以获取……，例如：
```
当用户询问收入增长、ARPU（用户平均收入）或收入构成相关问题时：加载 `references/revenue.md` 以获取计算公式和行业基准数据
```

#### 500 行法则

SKILL.md 整体控制在 500 行以内，确保 frontmatter 与正文能被完整注入上下文而不溢出。超过500行，通常需要重构
- 单一职责，多职责拆分成多个 skill
- 对文档拆分，将不同类型的文件放到不同的目录中，例如：参考文档到 references、计算放到 scripts、输出模板放到 templates、示例放到 examples

## 四、调用方式

1. **显式调用**：用户在输入框中输入 `/<skill-name>`（例如 `/review`、`/init`），系统识别后执行。
2. **隐式匹配**：skill 的描述会作为系统提示的一部分注入 LLM 上下文；当用户的请求与某个 skill 的描述高度吻合时，LLM 也会主动调用。
3. **硬性约束**：系统提示明确要求 *"Only use skills listed in the user-invocable skills section — don't guess"*。自定义的"伪 skill"不会被识别。

调用通过 `Skill` 工具完成，参数为 `skill`（技能名）和可选的 `args`（附加参数）。

## 五、内置技能清单（12 个）

按"配置 → 评审 → 执行 → 编程"的工作流排序。

### 1. `/update-config` —— 配置 Claude Code 自身

- **作用**：通过 `settings.json` 配置 Claude Code 的行为、钩子、权限、环境变量等。
- **适用场景**：
  - 添加自动行为（"从此以后每次 X 都 Y"这类规则必须落库到钩子，记忆和口头告知无法保证自动触发）
  - 调整权限白名单（"允许 npm 命令"）
  - 调试已有钩子、修改环境变量
- **关联**：与 `记忆` 形成"配置 + 知识"的双层机制；记忆承载语义规则，配置承载自动化行为。

### 2. `/keybindings-help` —— 自定义键盘快捷键

- **作用**：在 `~/.claude/keybindings.json` 中重绑按键、添加组合键。
- **适用场景**：
  - 把 `Ctrl+S` 改成"保存并提交"
  - 添加 chord（多键组合）快捷键
  - 修改提交键（Enter 之外的其他触发方式）
- **典型输入**：`/keybindings-help 重新绑定 Ctrl+S`

### 3. `/verify` —— 验证代码改动有效

- **作用**：实际跑应用、看行为，确认改动达到预期。
- **适用场景**：
  - 修了 bug 但只跑了类型检查，想看运行时是否真的修好
  - 提交前需要跑通完整流程
  - 用户在 PR 描述中要求"附验证截图/日志"
- **与 `/run` 的差异**：`/run` 强调"启动应用并截图"；`/verify` 强调"针对某个具体改动验证它有效"。

### 4. `/code-review` —— 评审当前 diff

- **作用**：评审工作区未提交改动，输出正确性、可简化、效率等问题。
- **参数**：
  - `--comment`：把发现写到 PR 内联评论
  - `--fix`：把发现直接应用到工作区
- **强度档位**：低/中（少而高置信）/ 高→最高（覆盖广，可能含不确定项）。
- **适用场景**：
  - 提交前自查
  - 大型 PR 的快速初审
  - 想让 AI 帮忙改而不是只看时用 `--fix`

### 5. `/simplify` —— 等价于 `/code-review --fix`

- **作用**：直接评审 diff 并把建议落地。**不**输出"建议清单让你自己改"，而是改完给你看。
- **适用场景**：
  - 信任 AI 的判断，希望一次到位
  - 重构完顺手清理

### 6. `/fewer-permission-prompts` —— 减少权限弹窗

- **作用**：扫描历史 transcript，找出反复出现且只读的命令（如 `ls`、`cat`、Glob/Grep），写入项目 `.claude/settings.json` 的 allowlist。
- **适用场景**：
  - 每次跑 `npm test` 都要确认权限
  - 长期在某个项目工作，想把"必然要跑的命令"提前放行
- **注意**：只会加**只读**或**经过筛选**的命令，破坏性命令不会被自动加。

### 7. `/loop` —— 周期性任务

- **作用**：以固定间隔重复执行某个 prompt 或斜杠命令。
- **典型用法**：`/loop 5m /babysit-prs`（每 5 分钟跑一次 PR 看护）
- **不适用**：一次性任务、有外部事件驱动的任务（这些用 `CronCreate` 更合适）。
- **调度规范**：
  - 避开整点（`:00`、`:30`），否则全世界的请求会同时打到后端
  - 7 天后自动过期
  - 若要"任务在重启后仍存活"，需要 `durable: true`（写到 `.claude/scheduled_tasks.json`）

### 8. `/claude-api` —— 构建 Claude API / SDK 应用

- **作用**：为使用 Anthropic SDK / Claude API 的项目提供专门技能，含 prompt caching、模型迁移、批处理、工具使用、流式输出等指导。
- **触发条件**（来自技能描述）：
  - 代码中出现 `import anthropic` 或 `@anthropic-ai/sdk`
  - 用户在写 Claude 应用
  - 涉及 prompt 缓存命中率、调优、上下文压缩、模型升级（4.5 → 4.6 → 4.7）
- **明确不触发**：`openai` / 其他厂商 SDK、不含 `anthropic` 的通用 ML 代码。
- **典型使用**：
  - 写新功能时让 AI 帮忙
  - 调 prompt 缓存（5 分钟 TTL、读写分离）
  - 把 Sonnet 4.6 的代码迁移到 Opus 4.7

### 9. `/run` —— 启动并验证应用

- **作用**：拉起项目应用并确认能跑起来。
- **典型工作流**：
  1. 找是否有项目自带的"启动"技能
  2. 没有则按项目类型（CLI / server / TUI / Electron / 浏览器）选用内置模板
  3. 启动后实际交互（截图、看日志、点按钮）
- **与 `/verify` 的差异**：`/run` 偏"启动并展示"；`/verify` 偏"针对改动验证"。

### 10. `/init` —— 初始化 CLAUDE.md

- **作用**：扫描当前代码库，生成一份适合本项目的 `CLAUDE.md`（含架构、命令、约定、关键文件）。
- **适用场景**：
  - 新接手一个项目，想快速给 AI 配上下文
  - 项目结构变了想刷新文档
- **生成后**：建议人工 review 一遍再提交，因为自动生成的 CLAUDE.md 可能漏掉团队隐性约定。

### 11. `/review` —— 评审 PR

- **作用**：评审一个已存在的 GitHub PR。
- **与 `/code-review` 的差异**：
  - `/code-review` 看工作区未提交改动
  - `/review` 看远端 PR（用 `gh` 拉评论、diff、状态）
- **典型工作流**：
  1. 拉 PR 描述、diff、CI 状态
  2. 输出评审意见清单
  3. （可选）把评审意见回写到 PR 评论

### 12. `/security-review` —— 安全评审

- **作用**：对当前分支未提交改动做 OWASP top 10 类别的安全扫描（注入、XSS、敏感数据泄露、依赖漏洞等）。
- **适用场景**：
  - 涉及认证、支付、文件上传、外部输入
  - 合规要求（PCI、SOC2）
  - 合并到 main 前的最后一道关卡
- **定位**：和 `/code-review` 是不同维度——`/code-review` 看正确性，`/security-review` 看安全性。

## 六、按场景的推荐组合

| 我想要…… | 推荐组合 |
| --- | --- |
| 给项目接入自动行为 | `/update-config`（写 settings.json） |
| 重构完想清干净 | `/simplify`（= `/code-review --fix`） |
| 修完 bug 想确认 | `/verify` 或 `/run`（视是否要启动应用） |
| 提交前自查 | `/code-review`（看清单）→ `/security-review`（扫安全）→ `/fewer-permission-prompts`（清弹窗） |
| 写 Claude API 应用 | `/claude-api` |
| 周期性巡检 | `/loop` |
| 给 PR 评审批意见 | `/review`（看 PR）→ `/code-review`（看本地未提交） |
| 项目首日 | `/init`（生成 CLAUDE.md） |
